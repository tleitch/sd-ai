Running the SD-AI Benchmark Locally

This guide explains how to benchmark local LLMs against the SD-AI evaluation suites and submit your results. All local model runs require Apple Silicon (M1 Pro or later). Cloud model runs work on any platform.

1 Hardware Requirements

Minimum (small models ≤70B)

  • Apple M1 Pro / M2 / M3 with ≥32 GB unified memory
  • 100 GB free disk space
  • macOS 14 Sonoma or later
  • Expected: ~3–8 s per request (70B Q4)

Recommended (400–700B models)

  • Apple M3 Ultra / M4 Ultra with ≥192 GB unified memory
  • 512 GB for full 671B models (MLX or GGUF)
  • Fast NVMe volume for models that don't fit in RAM
  • Our reference: Mac Studio M3 Ultra 512 GB
Memory rule of thumb: A 671B Q4_K_M model requires ~390 GB of unified memory to fit entirely on-chip. If it partially offloads to system RAM or disk, expect dramatically slower inference (10–100×). For MLX 4-bit, ~380 GB is needed. The KV cache adds 2–20 GB depending on context length.

C Start the local inference server ›

Option 1: LM Studio local server (easiest)

Load your model in LM Studio, go to Local Server, choose port (e.g. 1234), and click Start.

Option 2: mlx_lm directly (lower overhead, recommended for MLX models)

# Install
pip install mlx-lm

# Start server (replace path and port as needed)
python3 -m mlx_lm.server \
  --model /path/to/your/model \
  --port 8080 \
  --host 0.0.0.0 \
  --chat-template-args '{"thinking": false}'

The thinking: false flag disables reasoning mode on models that support it (e.g. Kimi K2.5, Qwen 3). Benchmarks measure task performance, not chain-of-thought quality.

Verify the server is running

curl -s http://localhost:8080/v1/models | python3 -m json.tool
D Create a model profile ›

Create evals/model-profiles/my-model.json. Copy a similar profile from the fork's model-profiles directory as a starting point.

{
  "alias": "My-Model-Q4",
  "lmStudioModelId": "author/model-name",
  "baseURL": "http://localhost:8080/v1",
  "quantization": "Q4_K_M",
  "parameterCount": "70B",
  "inferenceBackend": "llama.cpp",
  "contextLoaded": 32768,
  "kvCacheQuant": "off",
  "maxTokens": 4000,
  "timeoutMinutes": 10,
  "lenientJsonParsing": true,
  "jsonObjectMode": false,
  "structuredOutput": false
}
Key fields:
  • structuredOutput: false — required for mlx_lm (it ignores response_format; the harness handles JSON injection automatically)
  • lenientJsonParsing: true — allows recovery from extra text wrapping JSON
  • contextLoaded — actual context you loaded in LM Studio, not the model's max

3 Running the Benchmark

Create a run config evals/run-configs/my-model-cld.json:

{
  "modelProfile": "evals/model-profiles/my-model.json",
  "baseExperiment": "evals/experiments/leaderboardCLD.json",
  "engines": [
    {
      "name": "qualitative",
      "limits": { "tokensPerMinute": 10000, "requestsPerMinute": 4, "baselineTokenUsage": 6000 }
    },
    {
      "name": "qualitative-zero",
      "limits": { "tokensPerMinute": 10000, "requestsPerMinute": 4, "baselineTokenUsage": 6000 }
    }
  ],
  "seed": 4242,
  "concurrency": 1,
  "sequential": true,
  "leaderboard": "CLD",
  "variations": [
    { "temperature": 0, "top_p": 1.0, "top_k": -1 },
    { "temperature": 0.1, "top_p": 0.95, "top_k": -1 }
  ]
}

CLD Extraction leaderboard

npm run evals -- \
  --config evals/run-configs/my-model-cld.json \
  --output evals/results/my-model-cld.json

Discussion leaderboard

npm run evals -- \
  --config evals/run-configs/my-model-discuss.json \
  --output evals/results/my-model-discuss.json

Use baseExperiment: "evals/experiments/leaderboardDiscuss.json"

Expected runtime: CLD suite ≈ 53 tests × 2 engines × N variations. At 30 s/request (671B model), allow 1–3 hours per variation. Use sequential: true and concurrency: 1 for local models to avoid Metal GPU contention.

! Common Pitfalls

🚫

mlx_lm ignores response_format: json_object

The mlx_lm server silently ignores response_format and returns free text. The SD-AI harness handles this automatically when structuredOutput: false is set in your model profile — it injects a JSON instruction into the system prompt. Do not set structuredOutput: true for mlx_lm models or responses will fail to parse.

⚠️

Metal GPU OOM on long prompts

Even with 512 GB unified memory, Apple's Metal GPU allocator will OOM on prompts longer than ~60–80k tokens. This is a GPU memory constraint, not system RAM. Set contextLoaded to a value the model can handle without OOM — 16k–32k is safe for most models on 192 GB hardware; 64k–128k requires 384 GB+. When in doubt, lower the context window first.

⚠️

Jinja2 template incompatibility with multi-turn messages

Many GGUF and MLX models ship Jinja2 chat templates that expect a single user message per turn, not consecutive user messages. The SD-AI harness collapses consecutive user messages automatically, but if you build custom prompts outside the harness you may get garbled outputs or "template error" responses. Always validate your prompt structure with a quick sanity test first.

⚠️

temperature: 0 is not truly deterministic on Apple Silicon

Apple's Metal compute introduces non-deterministic floating-point order, so temperature: 0 (greedy decoding) will produce slightly different outputs across runs. For reproducible comparisons, use the same seed value (we use 4242) and report results as an average over ≥3 seeds if you need confidence intervals.

💡

Thinking / reasoning mode inflates latency and may hurt task scores

Models with built-in chain-of-thought (Kimi K2.5, Qwen 3, DeepSeek R1) produce long <think> blocks before answering. These tokens are not counted in the benchmark score but do inflate average response time significantly. For CLD extraction and discussion tasks, we found reasoning mode hurts structured output compliance — disable it with {"thinking": false} in mlx_lm's --chat-template-args flag. LM Studio has a "Reasoning" toggle in model settings.

💡

LM Studio model ID vs. file path

When running via mlx_lm directly (not LM Studio), use the full file path to the model directory as the model ID in your API calls. LM Studio uses a short author/model-name slug. If you see model not found errors, verify what the server reports via GET /v1/models and use that exact string in lmStudioModelId.

💡

Rate limiting: match your hardware capacity

The tokensPerMinute and requestsPerMinute limits in your run config must be set conservatively for local models. A 671B model may only process 500–1500 tokens/min. Too-high limits cause request queuing, timeout cascades, and inflated latency numbers. Start with requestsPerMinute: 3 and tokensPerMinute: 8000, then raise if the model is consistently fast.

💡

KV cache quantization can degrade output quality

LM Studio's KV cache quantization (Q8_0, Q4_0) reduces memory use but can cause subtle accuracy regressions on structured extraction tasks. Our benchmarks use kvCacheQuant: "off" for apples-to-apples comparison. If you enable KV cache quant, note it in your submission.

🔍

Partial offload destroys performance

If a model does not fit entirely in unified memory, llama.cpp will offload layers to CPU. Even 5% CPU offload can slow inference 5–20× on large models. Verify full GPU utilization with sudo powermetrics --samplers gpu_power -i 1000 and watch for "cpu offload" in llama.cpp startup logs. If offloading is unavoidable, report it in your submission.

5 Interpreting Results

Overall score = fraction of individual test cases passed (0–1). Higher is better.

CLD Extraction covers 53 test cases across four categories:

  • Qualitative Translation — extract causal relationships from English text
  • Causal Reasoning — identify causal structure in real-world contexts
  • Iteration — refine an existing model given feedback
  • Conformance — follow structural constraints and output format requirements

Discussion covers 21 test cases across three categories:

  • Feedback Explanation — correctly explain feedback loop dynamics
  • Model Building Steps — articulate sound modeling methodology
  • Error Fixing Suggestions — identify and fix formulation errors

Scores are not normalized for model size. A 671B model scoring 77% vs a cloud model scoring 88% does not mean the local model is "12% worse" in any practical sense — the gap must be weighed against hardware cost, privacy, latency, and throughput requirements.

6 Submitting Your Results

Submissions are PR-based — you add your actual results JSON to the repo and open a pull request. This means the numbers are always backed by real benchmark output that maintainers can verify before merging.

  1. 1

    Fork the repo and create a model profile

    Fork UB-IAD/sd-ai, then add a model profile JSON under evals/model-profiles/ as described in Step D above.

  2. 2

    Run the benchmark and save output

    npm run evals -- \
  --config evals/run-configs/my-model-cld.json \
  --output community-results/my-model-cld.json

    Save to community-results/ (this directory is gitignored in the main repo but not in forks).

  3. 3

    Add a submission metadata file

    Create community-results/my-model-cld.meta.json:

    {
  "contributor": "your-github-username",
  "hardware": "Apple M3 Max 128GB",
  "macOS": "15.3",
  "backend": "mlx_lm 0.22.0",
  "contextLoaded": 32768,
  "kvCacheQuant": "off",
  "thinkingMode": false,
  "fullGPU": true,
  "notes": "Any deviations from the standard setup"
}
  4. 4

    Open a pull request

    Commit both files and open a PR against UB-IAD/sd-ai. The maintainer will run npm run leaderboard from your branch to verify the scores before merging. Verified results are added to docs/data/leaderboard.json and the leaderboard page is updated.

Why PRs, not issues? Scores must be backed by a real results JSON file that the maintainer can inspect and re-verify. This prevents fabricated submissions and makes every leaderboard entry reproducible.
